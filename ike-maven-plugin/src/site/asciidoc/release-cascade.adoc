= The IKE Release Cascade
:icons: font
:source-highlighter: coderay
:toc: left
:toclevels: 3

The IKE foundation is three repositories that must release in
topological order, because each consumes the artifacts of the ones
above it:

----
ike-tooling  ‚Üí  ike-docs  ‚Üí  ike-platform
----

`ike-docs` and `ike-platform` declare `ike-maven-plugin` ‚Äî and
consume `ike-build-standards` ‚Äî through `${ike-tooling.version}`;
`ike-platform` also consumes `ike-docs` through `${ike-docs.version}`.
A downstream repo released against a stale upstream pin ships a
*split foundation*. The release cascade is the machinery that keeps
the three coherent.

== The goals

=== ike:release-publish ‚Äî one repo, upstream-aligned

Before cutting a release, `ike:release-publish` reads the repo's own
`release-cascade.yaml` and, for every `upstream` edge, resolves the
latest released version of that upstream and bumps the edge's
`version-property` when the POM is behind. A single-repo release is
therefore correct on its own ‚Äî it can never ship on a stale
foundation.

The preflight fails on any warning by default; pass
`-Dike.release.ignoreWarnings=true` to release past warnings
(errors are never ignorable). See `IKE-Network/ike-issues#428`.

=== ike:release-cascade ‚Äî the executor

Walks the whole cascade. Run from any foundation repo (with the
others checked out as siblings); it assembles the graph, walks it in
topological order, and runs `ike:release-publish` on every repo that
has unreleased changes ‚Äî skipping repos whose only commits since
their last tag are release-cadence bookkeeping.

[source,bash]
----
mvn ike:release-cascade                          # release the cascade
mvn ike:release-cascade -DpushRelease=false       # local-only walk
mvn ike:release-cascade -Dike.release.cascade.basedir=/path/to/checkouts
----

=== ike:cascade-export ‚Äî the topology

Assembles the graph and writes it as JSON or `.properties` ‚Äî the
input a CI meta-runner consumes to generate build-chain edges
instead of hand-wiring them (`IKE-Network/ike-issues#403`).
Read-only.

[source,bash]
----
mvn ike:cascade-export                            # JSON to stdout
mvn ike:cascade-export -Dformat=properties
mvn ike:cascade-export -DoutputFile=target/cascade.json
----

== Testing the cascade safely

Both inspection paths are side-effect-free, so the cascade can be
exercised at any time:

* `mvn ike:cascade-export` ‚Äî proves the per-project manifests
  assemble into a valid, ordered graph.
* `mvn ike:release-cascade -DpushRelease=false` ‚Äî walks every repo
  and reports its state. When the foundation is freshly released,
  every repo reports *up to date* and nothing is released:

== Running a real cascade

Check the three foundation repos out as siblings under one directory,
then run `ike:release-cascade` (or use the IntelliJ run configuration
of the same name) from any of them. The executor releases each repo
that has changes, in order, so each repo's Nexus deploy completes
before the next ‚Äî which `ike:release-publish` then aligns to the
just-released upstream ‚Äî begins.

== CI / TeamCity

The documented CI model runs one build configuration per foundation
repo, with TeamCity ‚Äî not `ike:release-cascade` ‚Äî orchestrating. The
build-chain edges are generated from `ike:cascade-export`; because
the assembler falls back to cloning from each edge's `url`, the
export runs on a bare CI agent that has only one repo checked out.
